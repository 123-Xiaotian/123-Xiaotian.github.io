设计概述

通过对于P4的verilog代码进行重构，本CPU实现了支持10条指令的流水线CPU，该CPU支持延迟槽，将指令统一阻塞在D级。

实现指令说明

R型指令

  • 算数/位运算指令：add,sub

I型指令

  • 算数/位运算指令： ori,lui
  • B类指令： beq,
  • 访存指令： lw,sw

J型指令

  • jal,jr

空指令

  • nop

数据冒险

阻塞

从D级开始，计算我们所需指令的T_use和T_new，T_use中不需要用到新数据的地方我们填入infty，T_new中不产生新数据的地方我们填入0。这里sw是唯一一个产生新数据但T_new=0的指令。

op T_use(rs) T_use(rt) T_new(D)
add/sub 1 1 2
sw 1 2 0
lw 1 infty 3
beq 0 0 0
lui infty infty 2
ori 1 infty 2
jal infty infty 1
jr 0 infty 0
nop infty infty 0

为正确阻塞，我们通过比较各级A3是否与D级指令相等且不为0，然后将T_use与T_new进行比较，将各级的rs，rt阻塞结果或起来，一旦有任意以及不满足条件就阻塞，实现了应阻塞尽阻塞

转发

在D,E,M级，我们分别设置了从E到D，从M到D，从W到D和从M到E，从W到E和从W到M级的转发，我们分别给出E,M，W级时，各个指令的T_new情况,不产生新数据的指令省略

op T_new(E) T_new(M) T_new(W)
add/sub 1 0 0
lw 2 1 0
lui 1 0 0
ori 1 0 0
jal 0 0 0

由表可知，W级的所有结果都是可以被各级接受的，在D级，我们能接受来自jal指令E级的结果，能接受jal指令和ALUrs在M级的结果以及所有指令来自W级的结果。在E级，能接受来自jal和ALUrs来及M级的结果和所有来自W级的结构，M级只能接受来自W级的结果。

功能模块定义

这里仅给出更改了模块内容的模块定义：

IFU（取指模块）

  • 端口定义
信号名 方向 位宽 描述
clk I 1 时钟信号
reset I 1 异步复位信号
stall I 1 阻塞信号
next I 32 下一条要被执行的指令的地址
pc O 32 正在执行的指令的地址
Instr O 32 正在执行的指令
  • 功能定义
序号 功能名称 功能描述
1 复位 reset信号有效时，将PC寄存器中的值置为0x00003000
2 阻塞 stall信号有效时，PC寄存器忽略时钟输入，PC当前值不变
3 写PC寄存器 当stop信号失效且clk上升沿来临时，将下一条指令的地址写入PC寄存器
4 取指令 根据当前PC的值从ROM(IM)中读出对应的指令到Instr端口

ALU (算数逻辑单元)

该模块对各个命令进行相应的计算，并将计算结果输出（删除了equal比较结果进行分支早判）

  • 端口定义
信号名 方向 位宽 描述
num1 I 32 运算数1
num2 I 32 运算数2
ALUOp I 4 运算种类符
ALUrs O 32 运算结果
  • 功能定义
序号 功能名称 ALUop 功能描述
1 加 4’b0000 ALUrs = num1 + num2
2 减 4’b0001 ALUrs = num1 - num2
3 或 4’b0010 ALUrs = num1 | num2
4 lui 4’b0011 ALUrs = num2 << 16

Controller (控制器)

本模块中，我们通过Add逻辑对OpCode和Funct进行解码，并通过或逻辑输出了RegDst等共9条控制指令，同时输出了ALUOp作为ALU运算种类的选择。增加了rsT_use,rtT_use,T_new指令的输出，便于阻塞逻辑判断。

  • 端口定义
信号名 方向 位宽 描述
funct I 6 funct域
op I 6 opcode域
RegDst O 1 0:grf从rt写入 1:grf从rd写入
ALUSrc O 1 ALU的num2来自 0:RD2 1:扩展后的立即数
RegWrite O 1 grf写使能信号
ExtOp O 1 Ext的扩展类型 0:0扩展 1:符号扩展
MemToReg O 1 grf的写入数据来自 0：ALU计算结果 1：Mem
MemWrite O 1 Mem写使能信号
Branch O 1 1代表是分支指令
Jal O 1 Jal指令信号
Jr O 1 Jr指令信号
ALUOp O 4 ALU操作数
rsT_use O 2 该指令最早经过几拍需要使用rs寄存器
rtT_use O 2 该指令最早经过几拍需要使用rt寄存器
T_new O 2 该指令最早经过几拍产生新结果
  • 控制信号定义
信号名 触发指令 描述
RegDst add,sub 0:grf从rt写入 1:grf从rd写入
ALUSrc ori,lw,sw,lui ALU的num2来自 0:RD2 1:扩展后的立即数
RegWrite add,sub,ori,lw,lui grf写使能信号
ExtOp lw,sw,beq Ext的扩展类型 0:0扩展 1:符号扩展
MToR lw grf的写入数据来自 0：ALU计算结果 1：Mem
MemWrite sw em写使能信号
Branch beq 1代表是分支指令
Jal jal 1表示grf写入31号寄存器，写入内容来自pc，同时1表示NPC的next来自Addr
Jr jr 1表示NPC的next来自rd1读出的Addr

NPC (指令转移器)

该模块实现了对于PC指令转移的的逻辑，针对一般指令和B类指令设置了不同的计算逻辑。并通过外部Branch信号选择NPC的值。由于NPC位于D级会导致所有指令执行两拍，我同时将IF_pc和ID_pc传入该模块进行计算。

  • 端口定义
信号名 方向 位宽 描述
PC I 32 IF_pc
ID_pc I 32 ID_pc
offset I 32 扩展后的地址偏移
JAddr I 32 J指令跳转地址
Branch I 1 0:PC+4 1：PC+4+偏移
Jal I 1 NPC来自JAddr
Jr I 1 NPC来自JAddr
NPC O 32 PC下一条指令地址
  • 功能定义
序号 功能名称 功能描述
1 指令转移 对指令地址进行计算并转移

思考题

Q：我们使用提前分支判断的方法尽早产生结果来减少因不确定而带来的开销，但实际上这种方法并非总能提高效率，请从流水线冒险的角度思考其原因并给出一个指令序列的例子。

A:分支早判将分支判断提早到D级，这一操作导致分支指令进入D级后立刻就需要使用rs，rt级的数据，这导致一旦分支指令与前序lw等需要过2~3拍才能产生新数据的指令产生数据冲突的时候，就一定会导致阻塞

如上述指令所示，由于分支早判，运行到beq的时候必定会产生一拍的阻塞，但如果beq在E级执行，则无需阻塞，当然，在实现延迟槽以后，D级判断可以保证没有需要冲刷的指令，但如果出现若干个上述指令块且每一个指令块都无需跳转，D级判断就会产生很多阻塞开销，

Q：因为延迟槽的存在，对于 jal 等需要将指令地址写入寄存器的指令，要写回 PC + 8，请思考为什么这样设计？

A：延迟槽的存在，保证了jal指令后的延迟槽指令一定会被执行，所以用jr跳转回后，在程序员视角，延迟槽指令不应该再次被执行，所以应当跳转到jal后的第2条指令的地址，也即pc+8

Q:我们要求所有转发数据都来源于流水寄存器而不能是功能部件（如 DM、ALU），请思考为什么？

A：首先，虽然流水线寄存器间的计算如ALU，Branch判断，Jal指令地址计算等操作都是组合逻辑，但实际上这个计算还是有一定的时间延迟的，假如将ALU指令的输出结果直接转发到D级，那么相当于一个时钟周期内，需要等ALU计算和后续计算这两个计算过程，这样就必定会导致流水线的时钟周期的延长，这种开销最终会作用在所有模块的计算中。其次，当我们需要M级到E级的转发的时候，如果将本来来及M级寄存器的数据换成来自ALU的数据，相当于从ALU的结果拿过来作为ALU的输入，这会导致电路的震荡而无法产生正确的结果

Q:我们为什么要使用 GPR 内部转发？该如何实现？

前一条指令（如 ALU 运算）需写入 GPR，后一条指令要读取同一个 GPR。无转发时，后指令需等待前指令完成写回才能取到正确数据，会插入1拍的阻塞。转发可直接将前指令的运算结果提前转发给D级，无需等待写回，消除阻塞。

实现方式：在我的架构中，只需要先对比ID_rs,ID_rt与WB_A3是否相等且不为0，如果满足就把ID_rd1/ID_rd2赋值为WD_WB即可。不过我把内部转发实际写在了D级而不是WB级，这样可以和其他转发放在一起，看起来条理更清楚一点。

GPR内部转发可以直接将来自WB级的指令转发到D级，由于WB级的所有指令的T_new均已为0，所以可以放心转发，在实现这个转发以后，可以保证

Q:我们转发时数据的需求者和供给者可能来源于哪些位置？共有哪些转发数据通路？

A:需求者：D级，E级和M级

供给者：E级，M级和W级

转发数据通路：从E级到D级，从M级到D级，从W级到D级和从M级到E级，从W级到E级和从W级到M级。

Q:在课上测试时，我们需要你现场实现新的指令，对于这些新的指令，你可能需要在原有的数据通路上做哪些扩展或修改？提示：你可以对指令进行分类，思考每一类指令可能修改或扩展哪些位置。

A：R型指令：controller中的ALUOp，ALU中的计算方式，可能要扩展shift接口实现数据左移，右移等逻辑
I型指令：B类指令需要扩展branch模块，controller模块中可能需要增加BranchOp接口，如果涉及字节的写入或加载，可能需要扩展DM的输入地址接口并增加内部的字节分支写入或读出判断。并需要增加接口判断读出或写入的是是w还是b。另外，因为目前实现的所有指令都没有在M级前用到内存中内容的场景，实际上没有涉及sw指令的任何转发，但是sw某种意义上是会产生新数据的
J型指令：需要修改Jaddr模块的计算逻辑

Q:简要描述你的译码器架构，并思考该架构的优势以及不足。

A:我的译码器采用控制信号驱动型的译码风格，主要分为add逻辑确定指令种类，or逻辑确定指令对应的控制信号，并最后通过组合逻辑确定每一个指令的rsT_use,rtT_use和T_new。该架构的优势主要在于代码量较小，且实现逻辑清晰。但是由于没有将每一个指令的控制信号独立出来，导致难以直观的确定每一个指令的控制信号，可能会导致控制信号的疏漏。

Q:[P5 选做] 请详细描述你的测试方案及测试数据构造策略。
A:在之前单一种类测试代码的基础上，我添加了指令联动代码测试所有转发逻辑的运行的正确性以及是否正确阻塞。数值计算仍采取边界数据测试，跳转指令测试了像前，后，的跳转测试，读写内存的偏移也测试了所有正负情况，联动代码中，针对不同的转发，适时地插入空泡和其他指令测试延迟槽的运行正确性
